Ecosystem Architecture

FitSaaS operates as a decoupled multi-service project. The client runs on Next.js, coordinating security tokens with a Fastify REST service backend mapping to a serverless Postgres database.

Our database consists of 5 central tables designed with high transactional consistency and automated cascade deletion rules:

  • User Table: Securely holds credentials (hashed), target weights, height, personalized water/calorie budgets, and cycle configurations.
  • Workout Table: Records individual exercise sessions, links directly to the User, and logs duration and optional notes.
  • Exercise Table: Logs specific movements inside a workout. Cascades on workout deletion.
  • Set Table: Individual physical calibrations containing reps and weight logged in kilograms.
  • MenstrualLog Table: Records cycle dates, symptomatology tags, mood levels, and energy parameters.

For detailed schema diagrams, data models, and enums, view our Database & API specifications sheet.

Core Features & How-To Guides

FitSaaS houses highly customized interfaces designed to facilitate daily wellness logging:

The Interface: On the Workouts panel, users can view predefined movement cards. Clicking 'Record Workout' executes a hardware-accelerated 3D Y-axis flip, seamlessly displaying a logging form.

Steps to Log a Workout:

  1. Go to the **Workouts** tab in your dashboard.
  2. Click the card representing the exercise you trained.
  3. Enter sets, repetitions, and weights used (in kg).
  4. Toggle 'Trained Lighter' if you are feeling fatigued during a cycle phase to record an adjusted performance milestone.
  5. Click **Save** — the card will flip back, committing the records instantly to PostgreSQL via Fastify.

Unlike traditional tracking tools, FitSaaS maps active training targets to natural hormonal rhythms:

  • Menstrual Phase (Days 1-5): High fatigue. Recommends light restorative yoga, stretching, and loads adjusted lower by ~20%.
  • Follicular Phase (Days 6-12): Rising energy. Perfect for building cardiorespiratory endurance and starting basic lifts.
  • Ovulatory Phase (Days 13-15): Peak strength. Recommends maximizing weight capacities, record-breaking sets, and high-intensity logs.
  • Luteal Phase (Days 16-28): Gradual recovery. Shifts focus toward steady-state strength training and mental focus.

Medical Subtypes: Checking the "Medical History" configuration in the profile opens up specialized diets for specific health situations, including:

  • Insulin-Resistant PCOS: High-protein, complex fiber meal structures to control insulin spikes.
  • IBS (Inflammatory Bowel): Tailored High/Low FODMAP structures preventing digestive distress.
  • Hypo/Hyperthyroidism: Iodine, selenium, and energy-adjusted diets.

Indian Protein customizer: Receive instant local Indian meal log variations (Paneer, Chana, Dal, Eggs, Fish, Chicken) customized to fit Vegetarian, Vegan, or Non-Vegetarian thresholds with single-click Lactose-Free/Gluten-Free filtration!

Privacy & Security Standards

FitSaaS is built with extreme security guidelines, protecting critical records:

  • Cryptographic Password Hashing: FitSaaS never stores passwords in plain text. Your credentials are encrypted using cryptographic hashing before saving to PostgreSQL.
  • Encrypted JWT Tokens: Session cookies generated by NextAuth are encrypted JSON Web Tokens (JWT) stored in secure browser cookies, shielding users against session Hijacking.
  • Zero Data Selling: Your physiological logs, menstrual calendars, and exercises are stored in isolated tables and never aggregated or shared with external trackers.

FAQ & Production Stability

Common troubleshooting topics and active architectural issues encountered during production deployment:

The Problem (Render Cold Start): To host FitSaaS 100% free of charge, our Fastify API server resides on a **Render Free container**. Render hibernates containers after 15 minutes of inactivity. Waking up a hibernated container takes **35–45 seconds**. NextAuth has a strict built-in sync timeout limit of **10 seconds**. When logging in after a break:

  1. Attempt #1: Render container cold start is initiated. NextAuth times out waiting, does not commit the authentication cookie, and redirects you back to `/login`.
  2. Attempt #2: Render server finishes booting up and user lookup completes in PostgreSQL.
  3. Attempt #3: Render is completely awake! Auth sync resolves in less than 150ms, committing the session cookie instantly and redirecting you to `/dashboard`.

  • For a detailed timing sequence flow diagram and fix options, view the Production Sign-In RCA Log.
